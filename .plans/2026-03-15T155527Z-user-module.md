assumptions:
- Users self-register; registrants receive ROLE_USER automatically
- ROLE_ADMIN is assigned by other admins via the admin UI (or seeded via DB for the first admin)
- Admin operations: list users, lock/unlock, change role, delete
- Admin UI uses HTMX-enhanced Thymeleaf (partial renders, no full page reloads)
- Deleting an admin's own account is blocked server-side
- Passwords stored as BCrypt hashes
- jOOQ generated classes used for DB access (via testcontainers-jooq-codegen-maven-plugin, same as order/inventory)
- Each module owns a dedicated PostgreSQL schema matching its name; "user" is a reserved word → always quoted as "user"
- Generated jOOQ classes land in target/generated-sources/jooq under simple.simple_webapp.user package
- SecurityConfiguration lives in the user module as a public @Configuration
- Thymeleaf + HTMX is the view layer (custom login/register pages)
- Spring Modulith module detection: last package segment = "user" → migrations in db/migration/user/
- No email verification in scope

  - phase: 1 — Database schema
    files:
      - src/main/resources/db/migration/user/V1__init.sql
    changes:
      - CREATE SCHEMA IF NOT EXISTS "user"
      - CREATE TABLE "user".users:
          id                      UUID PRIMARY KEY
          username                VARCHAR(255) UNIQUE NOT NULL
          password                VARCHAR(255) NOT NULL
          -- UserDetails flags (all default TRUE = fully active account)
          enabled                 BOOLEAN NOT NULL DEFAULT TRUE
          account_non_expired     BOOLEAN NOT NULL DEFAULT TRUE
          account_non_locked      BOOLEAN NOT NULL DEFAULT TRUE
          credentials_non_expired BOOLEAN NOT NULL DEFAULT TRUE
          -- Login failure tracking (for auto-lock support)
          failed_login_attempts   INT NOT NULL DEFAULT 0
          locked_at               TIMESTAMP WITH TIME ZONE NULL
      - CREATE TABLE "user".user_roles (user_id UUID FK → "user".users.id, role VARCHAR(50) NOT NULL, PK(user_id, role))
      - Note: schema name quoted everywhere ("user") because it is a PostgreSQL reserved word
      - Note: account_non_locked set FALSE + locked_at populated when locking an account

  - phase: 2 — Domain & service
    files:
      - src/main/java/simple/simple_webapp/user/package-info.java
      - src/main/java/simple/simple_webapp/user/UserRole.java
      - src/main/java/simple/simple_webapp/user/UserManagement.java
    changes:
      - package-info.java: @NullMarked (matches order/inventory pattern)
      - UserRole.java: public enum { USER, ADMIN }
      - UserManagement.java: public @Service implementing UserDetailsService
          - DSLContext + PasswordEncoder injected via constructor
          - Uses generated Tables.USERS and Tables.USER_ROLES (from simple.simple_webapp.user.tables)
          - register(String username, String password): inserts into "user".users + "user".user_roles with ROLE_USER; throws DuplicateUsernameException if username taken
          - loadUserByUsername(String): joins users + user_roles; auto-unlocks if locked_at + 5min < NOW() (resets account_non_locked=TRUE, failed_login_attempts=0 in DB); maps all UserDetails flags; throws UsernameNotFoundException
          - recordFailedAttempt(String username): increments failed_login_attempts; if >= 3 sets account_non_locked=FALSE, locked_at=NOW()
          - recordSuccessfulLogin(String username): resets failed_login_attempts=0
          - findAll(): returns list of user summaries (id, username, roles, account_non_locked, enabled)
          - lockUser(UUID id) / unlockUser(UUID id): manual lock/unlock; unlockUser also resets failed_login_attempts=0
          - setRole(UUID id, UserRole role): replaces all roles for the user
          - deleteUser(UUID id, String currentUsername): deletes user + roles; throws if id matches current user
      - LoginAttemptListener.java: package-private @Component
          - @EventListener(AuthenticationFailureBadCredentialsEvent): calls UserManagement.recordFailedAttempt(username)
          - @EventListener(AuthenticationSuccessEvent): calls UserManagement.recordSuccessfulLogin(username)
          - Note: uses Spring Security's built-in application events (not Spring Modulith cross-module events; regular @EventListener is correct here)

  - phase: 3 — Security configuration
    files:
